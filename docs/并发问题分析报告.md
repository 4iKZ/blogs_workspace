#### 【P0-1】Redis 连接池配置缺失导致连接耗尽风险

**问题描述**:  
项目使用 `RedisTemplate` 进行大量 Redis 操作，但在 `RedisConfig.java` 中未配置连接池参数。在高并发场景下，可能导致 Redis 连接耗尽，引发系统雪崩。

**影响范围**:
- 所有使用 Redis 的功能模块
- 缓存系统
- 分布式锁
- 排行榜功能
- 浏览量统计

**复现步骤**:
1. 使用 JMeter 或 wrk 模拟 500+ 并发用户
2. 持续发送文章浏览、点赞、评论等请求
3. 观察 Redis 连接数和响应时间
4. 预计 30-60 秒后出现连接超时错误

**根本原因分析**:
```java
// RedisConfig.java - 缺少连接池配置
@Bean
public RedisTemplate<String, Object> redisTemplate() {
    RedisTemplate<String, Object> template = new RedisTemplate<>();
    template.setConnectionFactory(connectionFactory);
    // ❌ 未配置连接池参数
    return template;
}
```

**潜在风险评估**:
- **系统稳定性**: ⚠️⚠️⚠️ 连接耗尽可能导致整个系统不可用
- **数据完整性**: ⚠️⚠️ 部分写操作可能失败，导致数据丢失
- **用户体验**: ⚠️⚠️⚠️ 请求超时，页面加载失败
- **业务连续性**: ⚠️⚠️⚠️ 核心功能（登录、缓存）不可用

**建议解决方案**:

1. **立即修复** - 添加连接池配置:
```java
@Bean
public RedisConnectionFactory redisConnectionFactory() {
    RedisStandaloneConfiguration config = new RedisStandaloneConfiguration();
    config.setHostName("localhost");
    config.setPort(6379);
    config.setPassword(RedisPassword.of("your-password"));
    
    GenericObjectPoolConfig poolConfig = new GenericObjectPoolConfig();
    poolConfig.setMaxTotal(50);      // 最大连接数
    poolConfig.setMaxIdle(30);       // 最大空闲连接
    poolConfig.setMinIdle(10);       // 最小空闲连接
    poolConfig.setMaxWaitMillis(3000); // 连接超时时间 3 秒
    poolConfig.setTestOnBorrow(true);   // 借用连接时检测
    poolConfig.setTestWhileIdle(true);  // 空闲时检测
    
    LettuceConnectionFactory factory = new LettuceConnectionFactory(config, poolConfig);
    return factory;
}
```

2. **监控告警** - 添加 Redis 连接池监控:
```java
// 在 RedisConfig 中添加监控 Bean
@Bean
public GenericObjectPoolMXBean redisPoolMXBean(GenericObjectPoolConfig poolConfig) {
    return poolConfig.getPoolNameMXBean();
}
```

3. **降级策略** - 实现 Redis 故障降级:
```java
@Cacheable(value = "articles", unless = "#result == null")
public ArticleDTO getArticle(Long id) {
    try {
        return articleMapper.selectById(id);
    } catch (RedisConnectionFailureException e) {
        log.error("Redis 故障，降级到数据库查询", e);
        return articleMapper.selectById(id); // 直接查数据库
    }
}
```

**优先级评级**: 🔴 **P0 - 极高风险**

#### 【P0-2】分布式锁看门狗内存泄漏风险

**问题描述**:  
`RedisDistributedLock.java` 中的看门狗任务使用 `ConcurrentHashMap` 存储 `ScheduledFuture`，但在异常情况下可能导致任务无法取消，造成内存泄漏。

**影响范围**:
- 评论点赞/取消点赞
- 评论删除
- 文章点赞/取消点赞
- 文章收藏/取消收藏

**复现步骤**:
1. 模拟 Redis 网络抖动场景
2. 高频调用点赞、评论接口（1000 次/秒）
3. 在锁获取和释放过程中模拟 Redis 超时
4. 观察 JVM 内存增长情况
5. 持续运行 24 小时后检查 `watchdogFutures` Map 大小

**根本原因分析**:
```java
// RedisDistributedLock.java
private final ConcurrentHashMap<String, ScheduledFuture<?>> watchdogFutures = new ConcurrentHashMap<>();

private void startWatchdog(String lockKey, String lockValue, long expireTime, TimeUnit expireUnit) {
    ScheduledFuture<?> future = watchdogExecutor.scheduleAtFixedRate(() -> {
        try {
            Boolean exists = redisTemplate.hasKey(lockKey);
            if (Boolean.FALSE.equals(exists)) {
                cancelWatchdog(lockKey); // ✅ 正常路径
                return;
            }
            // ❌ 如果这里抛出异常，任务不会被取消
            Object currentValue = redisTemplate.opsForValue().get(lockKey);
            if (!lockValue.equals(currentValue)) {
                cancelWatchdog(lockKey);
                return;
            }
            redisTemplate.expire(lockKey, expireTime, expireUnit);
        } catch (Exception e) {
            log.error("看门狗续期失败，key: {}", lockKey, e);
            // ❌ 异常时没有取消任务，导致泄漏
        }
    }, delay, delay, TimeUnit.MILLISECONDS);
    
    watchdogFutures.put(lockKey, future);
}
```

**潜在风险评估**:
- **系统稳定性**: ⚠️⚠️⚠️ 内存泄漏最终导致 OOM
- **数据完整性**: ⚠️ 锁机制失效可能导致并发问题
- **用户体验**: ⚠️⚠️ 系统响应变慢直至崩溃
- **业务连续性**: ⚠️⚠️⚠️ 服务不可用

**建议解决方案**:

1. **修复看门狗异常处理**:
```java
private void startWatchdog(String lockKey, String lockValue, long expireTime, TimeUnit expireUnit) {
    long delay = expireUnit.toMillis(expireTime) / 3;
    if (delay < 100) {
        delay = 100;
    }

    ScheduledFuture<?> future = watchdogExecutor.scheduleAtFixedRate(() -> {
        boolean shouldCancel = false;
        try {
            Boolean exists = redisTemplate.hasKey(lockKey);
            if (Boolean.FALSE.equals(exists)) {
                shouldCancel = true;
                log.debug("看门狗取消：锁已不存在，key: {}", lockKey);
                return;
            }

            Object currentValue = redisTemplate.opsForValue().get(lockKey);
            if (!lockValue.equals(currentValue)) {
                shouldCancel = true;
                log.warn("看门狗取消：锁 value 不匹配，key: {}", lockKey);
                return;
            }

            redisTemplate.expire(lockKey, expireTime, expireUnit);
        } catch (Exception e) {
            log.error("看门狗续期失败，key: {}", lockKey, e);
            // ✅ 异常时也标记需要取消
            shouldCancel = true;
        } finally {
            // ✅ 确保任务被取消
            if (shouldCancel) {
                cancelWatchdog(lockKey);
            }
        }
    }, delay, delay, TimeUnit.MILLISECONDS);

    watchdogFutures.put(lockKey, future);
}
```

2. **添加定期清理机制**:
```java
@Scheduled(fixedRate = 300000) // 每 5 分钟清理一次
public void cleanupStaleWatchdogs() {
    long now = System.currentTimeMillis();
    watchdogFutures.entrySet().removeIf(entry -> {
        ScheduledFuture<?> future = entry.getValue();
        if (future.isDone() || future.isCancelled()) {
            log.debug("清理已完成的看门狗任务：{}", entry.getKey());
            return true;
        }
        return false;
    });
}
```

3. **设置最大任务数限制**:
```java
private static final int MAX_WATCHDOG_TASKS = 10000;

private void startWatchdog(...) {
    if (watchdogFutures.size() >= MAX_WATCHDOG_TASKS) {
        log.warn("看门狗任务数达到上限，强制清理旧任务");
        cleanupStaleWatchdogs();
    }
    // ...
}
```

**优先级评级**: 🔴 **P0 - 极高风险**

#### 【P1-1】缓存一致性验证的线程安全问题

**问题描述**:  
`CacheConsistencyVerifier.java` 中的定时验证任务在遍历缓存时没有加锁，可能导致并发修改异常或数据不一致。

**影响范围**:
- 点赞状态缓存验证
- 收藏状态缓存验证
- 所有使用定时验证的缓存

**复现步骤**:
1. 启动缓存一致性验证定时任务（每 5 分钟执行）
2. 在验证过程中，模拟高并发的点赞/收藏操作
3. 观察日志中的异常信息
4. 检查验证结果的准确性

**根本原因分析**:
```java
// CacheConsistencyVerifier.java
@Scheduled(cron = "0 */5 * * * ?")
public void verifyLikeStatusConsistency() {
    List<UserLike> likes = userLikeMapper.selectRecentRecords(sampleSize);
    
    for (UserLike like : likes) {
        String cacheKey = RedisCacheUtils.generateArticleLikeKey(
            like.getArticleId(), like.getUserId()
        );
        
        Boolean cached = (Boolean) redisCacheUtils.getCache(cacheKey);
        // ❌ 没有加锁，验证期间缓存可能被其他线程修改
        if (cached != null && cached != expected) {
            redisCacheUtils.deleteCache(cacheKey); // ❌ 可能误删正常缓存
        }
    }
}
```

**潜在风险评估**:
- **数据完整性**: ⚠️⚠️ 可能误删正常缓存
- **系统稳定性**: ⚠️ 并发修改可能导致异常
- **性能**: ⚠️⚠️ 验证期间影响正常业务操作

**建议解决方案**:

1. **使用乐观锁机制**:
```java
public void verifyLikeStatusConsistency() {
    List<UserLike> likes = userLikeMapper.selectRecentRecords(sampleSize);
    
    for (UserLike like : likes) {
        String cacheKey = RedisCacheUtils.generateArticleLikeKey(
            like.getArticleId(), like.getUserId()
        );
        
        // ✅ 先检查数据库中记录是否仍然存在
        UserLike dbRecord = userLikeMapper.findByUserIdAndArticleId(
            like.getUserId(), like.getArticleId()
        );
        
        if (dbRecord == null) {
            // 数据库记录已删除，清理缓存
            redisCacheUtils.deleteCache(cacheKey);
            log.debug("清理已删除记录的缓存：{}", cacheKey);
            continue;
        }
        
        // ✅ 只在缓存明确为 false 时才修正
        Boolean cached = (Boolean) redisCacheUtils.getCache(cacheKey);
        if (Boolean.FALSE.equals(cached)) {
            redisCacheUtils.deleteCache(cacheKey);
            log.warn("修正错误的缓存状态：{}", cacheKey);
        }
    }
}
```

2. **降低验证频率和范围**:
```yaml
# application.yml
cache:
  consistency:
    verification-interval-minutes: 30  # 从 5 分钟改为 30 分钟
    verification-sample-size: 50       # 从 100 减少到 50
```

**优先级评级**: 🟠 **P1 - 高风险**

#### 【P1-2】异步事件处理的错误恢复机制缺失

**问题描述**:  
系统大量使用 `@Async` 和 `ApplicationEventPublisher` 进行异步处理（如通知发送、缓存更新），但缺乏完善的错误恢复和重试机制。

**影响范围**:
- 通知发送
- 缓存失效
- 浏览量同步
- 排行榜更新

**复现步骤**:
1. 模拟 Redis 临时故障（5-10 秒）
2. 触发文章浏览、点赞、评论等操作
3. 观察异步事件的执行情况
4. 检查通知和缓存状态

**根本原因分析**:
```java
// CacheInvalidationListener.java
@Async
@EventListener
public void handleCacheInvalidation(CacheInvalidationEvent event) {
    try {
        // ... 处理逻辑
    } catch (Exception e) {
        log.error("缓存失效处理失败：{}", event, e);
        // ❌ 仅记录日志，没有重试机制
        // ❌ 事件丢失，缓存可能永久不一致
    }
}

// NotificationEventListener.java
@Async
@EventListener
public void handleNotification(NotificationEvent event) {
    try {
        notificationService.createNotification(...);
    } catch (Exception e) {
        log.error("创建通知失败：{}", event, e);
        // ❌ 通知丢失，用户可能收不到重要通知
    }
}
```

**潜在风险评估**:
- **数据完整性**: ⚠️⚠️ 缓存不一致、通知丢失
- **用户体验**: ⚠️⚠
