# 前端代码问题报告（2026-03-06）

## 1. 审查范围

本次主要审查 `frontend/` 目录下的 Vue 3 + Vite + Pinia 前端代码，重点覆盖：

- 工程配置：`package.json`、`vite.config.ts`、`tsconfig.json`
- 启动与初始化：`src/main.ts`、`src/App.vue`
- 路由与鉴权：`src/router/index.ts`
- 状态管理：`src/store/*.ts`
- 网络层与服务层：`src/utils/axios.ts`、`src/services/*.ts`
- 关键页面与组件：`src/views/*.vue`、`src/components/*.vue`
- 运行日志：`frontend/frontend.log`
- 后端接口契约交叉验证：`src/main/java/com/blog/controller/NotificationController.java`、`CommentController.java`

---

## 2. 执行摘要

当前前端代码整体功能面较全，但存在以下几个突出问题：

1. **初始化时序存在高风险缺陷**：`router` 在 `pinia` 之前挂载，而路由守卫和请求拦截器都直接访问 store，可能导致首次加载/刷新时鉴权异常。
2. **登录态与权限状态设计不稳**：令牌持久化到 `localStorage`，并且多个页面直接读写 `localStorage`，造成安全风险和状态一致性问题。
3. **若干界面存在明确功能 bug**：例如桌面端主题切换按钮实际不会切换主题、移动端退出登录按钮不会真正执行登出。
4. **接口契约与前端实现不一致**：评论列表分页 UI 与后端真实返回结构不一致，导致分页逻辑事实上不可用。
5. **代码组织存在明显技术债**：超大单文件、服务层被绕过、日志输出过多、缺少 lint/test 基线，后续维护成本高。

---

## 3. 问题清单（按优先级）

## P0 / 高优先级

### 3.1 路由守卫与请求拦截器在 Pinia 初始化前访问 store，存在启动/刷新异常风险

**证据**

- `frontend/src/main.ts:11-12`
  - 先执行 `app.use(router)`，后执行 `app.use(pinia)`
- `frontend/src/router/index.ts:48`
  - 路由守卫里直接调用 `useUserStore()`
- `frontend/src/utils/axios.ts:34,77,83`
  - axios 请求/响应拦截器中直接调用 `useUserStore()`

**影响**

- 首次启动或页面刷新时，路由守卫/请求拦截器可能在 Pinia 还没激活时访问 store。
- 典型风险包括：
  - 控制台报 `getActivePinia()` 相关错误；
  - 首屏导航异常；
  - 请求拦截器拿不到 token；
  - 刷新后偶发被重定向到错误页面。

**修复建议**

- 调整初始化顺序，先 `app.use(pinia)`，再 `app.use(router)`。
- 或者在路由守卫 / axios 拦截器中显式传入 Pinia 实例，而不是依赖当前激活上下文。

---

### 3.2 管理员权限判断依赖迟到初始化的 `userInfo`，刷新后台路由会误判

**证据**

- `frontend/src/store/user.ts:7-10`
  - 初始化时只从 `localStorage` 读取 `token` / `refreshToken` / `isLoggedIn`
  - `userInfo` 初始值仍是 `null`
- `frontend/src/router/index.ts:58`
  - 管理员校验依赖 `userStore.getRole !== 'admin'`
- `frontend/src/App.vue:18-19`
  - `userStore.initUserInfo()` 在应用渲染阶段才执行，晚于首次路由判断

**影响**

- 管理员刷新 `/admin` 页面时，即使 token 还在，`userInfo` 也可能尚未恢复，`getRole` 为 `undefined`。
- 最终表现为：**管理员刷新后台页被错误重定向到首页**。

**修复建议**

- 在 store 初始化阶段就恢复 `userInfo`。
- 或在路由守卫中先等待用户状态恢复完成后再做权限判断。
- 将“是否登录”和“用户角色”恢复逻辑统一放在 store 启动流程，而不是零散地在组件里调用 `initUserInfo()`。

---

### 3.3 认证令牌保存在 `localStorage`，且刷新令牌同样暴露，存在安全风险

**证据**

- `frontend/src/store/user.ts:8-10,25,32,37,52-54`
  - `token`、`refreshToken`、`userInfo` 全部保存在 `localStorage`
- 多处页面组件直接读写 `localStorage`：
  - `frontend/src/views/ProfileView.vue:797-805,893-898`
  - `frontend/src/views/UserProfileView.vue:177-180,259-260`
  - `frontend/src/views/ArticleDetailView.vue:312-324`

**影响**

- 一旦页面存在 XSS 漏洞，攻击者可直接读取 access token / refresh token。
- 刷新令牌长期暴露意味着攻击面比仅存 access token 更大。
- 状态被多个页面直接改写，也容易出现 store 与本地缓存不一致。

**修复建议**

- 优先采用 HttpOnly Cookie 存储 refresh token。
- access token 如需前端持有，至少减少直接散落在页面中的 `localStorage` 读写。
- 统一由 store 封装登录态持久化与恢复，组件只通过 store/action 访问。

### 3.4 桌面端主题切换按钮实际不会切换主题

**证据**

- `frontend/src/components/Header.vue:52-59`
  - 桌面按钮仅绑定 `@click="toggleTheme"`
- `frontend/src/components/Header.vue:450-459`
  - `toggleTheme()` 内没有反转 `isDark.value`
  - 注释写的是“`isDark` 已由 v-model 更新”，但桌面按钮不是 `v-model`
- 只有移动端 `el-switch` 使用了 `v-model="isDark"`：`frontend/src/components/Header.vue:343-346`

**影响**

- 桌面端点击主题按钮时，主题状态不会真正翻转。
- 用户会看到按钮存在，但点击后无效果，属于明确功能缺陷。

**修复建议**

- 在桌面按钮点击时先手动反转 `isDark.value`，再应用主题。
- 或拆成 `setTheme(theme)` 和 `toggleTheme()` 两个更清晰的函数，避免把“状态变更”和“DOM 应用”混在一起。

### 3.5 移动端“退出登录”按钮没有真正调用登出方法

**证据**

- `frontend/src/components/Header.vue:327-335`
  - 模板里写的是：
  - `handleLogout; mobileMenuOpen = false;`
  - 这里仅引用了函数，没有执行 `handleLogout()`

**影响**

- 移动端点击“退出登录”时，大概率只会关闭菜单，不会真正清理登录态。
- 属于用户可直接感知的功能错误。

**修复建议**

- 改为显式调用 `handleLogout()`。
- 建议不要在模板里写多条语句，封装成专门的 `handleMobileLogout()`，可读性更高。

## P1 / 中高优先级

### 3.6 评论分页 UI 与后端接口契约不一致，分页逻辑实际上不可用

**证据**

前端：
- `frontend/src/components/comment/CommentSection.vue:30-39`
  - 使用了 `el-pagination`，依赖 `total`
- `frontend/src/components/comment/CommentSection.vue:67,143,223`
  - `total` 被声明，但在加载成功后并未从响应中赋值
- `frontend/src/services/commentService.ts:9-10`
  - `getList()` 类型声明返回 `Comment[]`

后端：
- `src/main/java/com/blog/controller/CommentController.java:42-58`
  - 接口实际返回 `Result<List<CommentDTO>>`

**影响**

- 评论列表接口没有返回总数，前端分页器缺少必要数据来源。
- 当前实现下：
  - 分页器可能始终不显示；
  - `currentPage/pageSize` 变化没有可靠意义；
  - 用户以为支持分页，实际体验不完整。

**修复建议**

- 二选一：
  1. 后端改为返回 `PageResult<CommentDTO>`；
  2. 前端取消假分页，改成“加载更多”或无限滚动。
- 同时清理 `CommentSection.vue` 中未生效的 `total` 相关逻辑。

### 3.7 大量页面绕过 service 层直接调 axios，接口契约分散且难维护

**证据**

直接调用 axios 的页面包括但不限于：
- `frontend/src/views/UserProfileView.vue:187,199,213,227,259`
- `frontend/src/views/SearchView.vue:71`
- `frontend/src/views/ProfileView.vue:592,604,790,822,860,880,889`
- `frontend/src/views/CategoryView.vue:181`
- `frontend/src/views/ArticleEditView.vue:179,190,261,308,311`
- `frontend/src/views/ArticleDetailView.vue:245`

同时项目已经存在 service 层：
- `frontend/src/services/articleService.ts`
- `frontend/src/services/authService.ts`
- `frontend/src/services/categoryService.ts`
- `frontend/src/services/notificationService.ts`
- 等

**影响**

- API 逻辑散落在页面组件里，返回结构处理方式不统一。
- 当接口字段变化时，需要在多个页面重复修改。
- 不利于 mock、单测、重试、缓存和错误处理统一化。

**修复建议**

- 将页面中的直接请求逐步收敛到 service 层。
- service 层负责：参数拼装、返回结构归一、异常语义化。
- 页面层只保留 UI 交互与状态组合逻辑。

### 3.8 用户状态被多个页面直接读写 `localStorage`，容易造成状态漂移

**证据**

- `frontend/src/views/ProfileView.vue:797-805,893-898`
- `frontend/src/views/UserProfileView.vue:177-180,259-260`
- `frontend/src/views/ArticleDetailView.vue:312-324`
- `frontend/src/store/user.ts` 本身也维护同一份数据

**影响**

- 同一份用户资料有“store 内状态”和“页面直接写入 localStorage”两套来源。
- 很容易出现：
  - 页面 A 改了 `localStorage`，页面 B 仍在看旧 store；
  - 某些界面必须刷新才能看到最新昵称/头像/角色；
  - 权限逻辑与展示逻辑不一致。

**修复建议**

- 将 `userInfo` 的更新统一收口到 `userStore.setUserInfo()`。
- 页面层禁止直接改 `localStorage`，只调用 store action。
- `canManageArticle` 等逻辑改为直接依赖 store，而不是重新解析 `localStorage`。

### 3.9 `storage` 事件监听未清理，存在内存泄漏和重复监听风险

**证据**

- `frontend/src/views/ArticleEditView.vue:362-366`
  - 直接在模块顶层 `window.addEventListener('storage', ...)`
- `frontend/src/views/ArticleDetailView.vue:395-399`
  - 同样直接注册 `storage` 监听
- 这两个页面的卸载逻辑只清理了 `matchMedia` / `resize` / `scroll`，没有清理 `storage`

**影响**

- 页面每次进入都会新增一个监听器，离开页面后不会释放。
- 长时间使用后，主题切换等事件可能触发多次回调。
- 这类问题通常难定位，会表现为“偶发重复执行”。

**修复建议**

- 在 `onMounted` 中注册监听，在 `onBeforeUnmount/onUnmounted` 中移除。
- 抽成统一的主题 composable，例如 `useTheme()`，避免多个页面重复写监听代码。

### 3.10 首页 / 用户主页存在重复首屏请求

**证据**

#### 首页 `HomeView.vue`
- `frontend/src/views/HomeView.vue:158-169`
  - `watch(..., { immediate: true })` 触发一次 `getArticles()`
- `frontend/src/views/HomeView.vue:205-208`
  - `onMounted()` 又调用一次 `getArticles()`

#### 用户主页 `UserProfileView.vue`
- `frontend/src/views/UserProfileView.vue:320-342`
  - `watch(activeMainTab, ..., { immediate: true })`
- `frontend/src/views/UserProfileView.vue:344-351`
  - `onMounted()` 又请求用户信息和文章列表

**影响**

- 首屏会发送重复请求，浪费接口资源。
- 在弱网或高延迟环境下，可能出现 loading 抖动、列表覆盖、日志噪音。

**修复建议**

- 明确“初始化加载”只保留一处：`onMounted` 或 `immediate watch` 二选一。
- 对依赖路由参数的加载逻辑，优先集中在单一 `loadData()` 函数中处理。

### 3.11 通知下拉的多个标签页尚未实现，UI 已开放但行为是假的

**证据**

- `frontend/src/components/Header.vue:402-407`
  - `follow/message/system` 三类 tab 直接 `return false`
  - 代码中保留 `TODO`

**影响**

- 用户可以点击这些标签，但永远看不到任何数据。
- 会造成“功能看起来存在，实际上不可用”的感知落差。

**修复建议**

- 若后端暂未支持，先隐藏这些 tab。
- 若前端已有数据结构，补齐类型映射逻辑，不要让 UI 暴露未完成功能。

### 3.12 注册页头像上传流程设计不一致，可能导致未登录用户无法上传头像

**证据**

- `frontend/src/views/RegisterView.vue:68-75`
  - 头像上传直接请求 `/api/user/avatar/upload`
- `frontend/src/views/RegisterView.vue:287-295`
  - 请求头从 `localStorage` 读取 token 作为 Bearer Token

**影响**

- 注册场景下用户通常尚未登录，没有 token。
- 如果后端上传接口要求鉴权，则注册页头像上传会失败。
- 即使后端允许匿名上传，当前写法也在流程设计上不清晰。

**修复建议**

- 明确注册期头像上传策略：
  - 要么允许匿名临时上传；
  - 要么注册完成后再上传头像。
- 前端不要在注册页隐式依赖登录 token。

### 3.13 组件文件体积过大，可维护性明显下降

**证据**

- `frontend/src/components/Header.vue`：**1342 行**

- `frontend/src/views/ArticleDetailView.vue`：**764 行**
- `frontend/src/views/ArticleEditView.vue`：**563 行**

**影响**

- 单文件承担过多职责：展示、状态、网络、权限、主题、通知、上传、分页等混杂在一起。
- 修改一个点容易影响多个功能。
- 不利于单测、复用、代码审查与新人接手。

**修复建议**

- 以职责拆分：
  - `Header` 拆成搜索、通知、用户菜单、主题切换等子组件；
  - `ProfileView` 拆成资料卡、设置弹窗、文章列表、点赞/收藏列表、关注粉丝列表。
- 将通用逻辑提炼到 composables 中。

## P3 / 低优先级 / 技术债

### 3.17 `any` 使用较多，削弱 TypeScript 约束价值

**证据**

- `frontend/src/views/UserProfileView.vue:161,165,166,189,201,215,229,265,274`
- `frontend/src/views/SearchView.vue:50`
- `frontend/src/views/ProfileView.vue:478,489,509,568,606,637,666,680,694,735,749,809,836,863,874,912`
- `frontend/src/services/articleService.ts:58,65,68`
- 等

**影响**

- 类型系统形同虚设，接口字段变更时更容易出现运行时错误。

- 对 API 响应统一建模，不要在页面中临时猜测结构。

## 5. 建议的修复顺序

### 第一阶段：先修功能与稳定性
1. 修复 `Pinia` 初始化顺序问题。
2. 修复管理员刷新后台页误判。
3. 修复桌面主题切换无效。
4. 修复移动端退出登录按钮未调用函数。
5. 修复评论分页契约不一致问题。

### 第二阶段：收口状态与接口层
6. 统一用户状态读写入口，移除页面中直接操作 `localStorage` 的逻辑。
7. 将页面中的直接 `axios` 请求逐步迁移到 service 层。
8. 抽离主题、通知、用户资料等通用 composable。

### 第三阶段：治理技术债
9. 拆分超大组件。
10. 增加 ESLint / Prettier / Vitest。
11. 清理生产调试日志。
12. 减少 `any`，补齐类型。

---

## 6. 结论

这套前端代码当前**不是“不能用”**，但已经出现了比较典型的“功能能跑、结构失控”的信号：

- 初始化与鉴权链路脆弱；
- 状态源不统一；
- 部分 UI 有明确 bug；
